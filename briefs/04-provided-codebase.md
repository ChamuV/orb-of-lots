# Provided Codebase

## Overview

The starter code is provided as compiled class files rather than source code.

Unlike previous worksheets and coursework, the source code for the framework has not been provided.

Only the public APIs and documentation should be used.

> Do not attempt to reverse engineer the class files. This defeats the purpose of the exercise and provides no benefit for completing the coursework.

---

## Package Structure

The codebase contains four packages:

```text
game
gui
student
main
```

### game

Contains the core game logic and framework.

### gui

Contains the graphical user interface used for visualising the game.

### student

This is the package where all coursework code should be written.

Primary implementation target:

```java
student.Explorer
```

### main

Contains the program entry points used to launch the game.

---

## Running the Program

Two main classes are provided.

### Headless Mode

```java
main.TXTmain
```

Runs the game without a graphical interface.

Useful for:

- automated testing,
- benchmarking,
- running many simulations.

---

### GUI Mode

```java
main.GUImain
```

Runs the game with a visual display.

Useful for:

- debugging,
- visualising behaviour,
- understanding map traversal.

---

## Initial Behaviour

Before implementing any solution code:

- the explorer remains stationary,
- the program returns immediately from `explore()`,
- an exception or error message is produced indicating that the Orb was not found.

Running the starter code before development is recommended to confirm that:

- the project compiles,
- the runtime environment is configured correctly.

# Command Line Options

Two optional command-line flags are available.

---

## Number of Runs

```text
-n <count>
```

Runs the program multiple times.

Example:

```bash
java main.TXTmain -n 100
```

### Notes

- Available only in headless mode.
- Ignored when using the GUI.
- Outputs results for each run.
- Provides an average score at the end.

Useful for:

- comparing algorithms,
- measuring robustness,
- benchmarking performance.

---

## Fixed Seed

```text
-s <seed>
```

Runs the program using a predefined random seed.

Example:

```bash
java main.GUImain -s 1050
```

### Notes

- Available in both GUI and headless modes.
- Generates the same map every time.
- Useful for debugging difficult maps.
- Enables reproducible testing.

---

## Combining Flags

Flags may be combined.

Example:

```bash
java main.TXTmain -n 100 -s 1050
```

This runs the same map generated by seed `1050` one hundred times.

# Recommended Development Workflow

## Early Development

Use GUI mode:

```bash
java main.GUImain
```

Purpose:

- understand map structure,
- observe explorer behaviour,
- debug movement logic.

---

## Algorithm Testing

Use fixed seeds:

```bash
java main.GUImain -s 1050
```

Purpose:

- reproduce bugs,
- compare algorithm changes,
- test difficult scenarios.

---

## Performance Evaluation

Use headless mode with many runs:

```bash
java main.TXTmain -n 100
```

Purpose:

- evaluate average score,
- compare different search strategies,
- estimate robustness across random maps.
